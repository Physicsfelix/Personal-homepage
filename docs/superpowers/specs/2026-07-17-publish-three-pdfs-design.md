# 发布三份 PDF 到个人主页

## 背景

个人主页的 Notes 页面已经通过 `data/notes.yml` 生成 PDF 卡片，并使用 `files/notes/` 下的稳定文件地址提供下载。本次需要把《泛函分析 II：算子理论》的旧稿替换为 2026 年 7 月 17 日完成的新稿，同时新增两份丘成桐大学生数学竞赛非官方题解汇编。

## 目标

- 保持《泛函分析 II》现有下载地址不变，原位替换 PDF，避免旧链接失效。
- 将《泛函分析 II》的状态改为“已完成”，版本保持 `2026.07`，日期改为 `2026-07-17`，删除旧版未完成说明。
- 新增《丘成桐大学生数学竞赛数学物理历年题解（非官方）》和《丘成桐大学生数学竞赛概率与统计历年题解（非官方）》两个下载条目。
- 两份题解按封面标记为“校订稿”，版本为 `2026.07`，日期为 `2026-07-17`，并在条目描述中保留覆盖年份与非官方属性。
- 更新 README 的公开资料表和版本说明，使仓库说明与网页一致。
- 更新站点验证脚本中的允许文件、SHA-256、条目数量和状态数量，确保新增或替换后的内容是预期文件。
- 推送到 `main` 后由现有 GitHub Actions 发布到 GitHub Pages。

## 方案选择

采用“稳定链接原位替换并在现有列表新增卡片”的方案。它不改变页面结构，也不会破坏《泛函分析 II》的现有链接。未采用为新稿另起文件名的方案，因为这会产生不必要的链接迁移；未采用旧版归档方案，因为用户要求直接上传，且没有提出保留历史版本入口。

## 数据与发布流程

三份源 PDF 复制到 `files/notes/`，`data/notes.yml` 提供卡片元数据，Quarto 构建 Notes 页面并复制 PDF 到发布目录。验证脚本检查文件哈希、卡片数、下载链接数和构建产物。提交推送到 `main` 后，现有发布工作流负责部署。

## 错误处理与验证

- 在替换前确认三份 PDF 可打开、未加密、页数正常，且封面与元数据一致。
- 使用 SHA-256 固定本次发布的三份文件，防止复制错误。
- 运行 JavaScript 下载功能测试、主题/内容验证脚本和 Quarto 完整渲染。
- 检查生成的 Notes 页面包含 6 张卡片和 6 个唯一 PDF 链接。
- 推送后检查 GitHub Actions 发布结果和线上三个下载地址；若发布失败，保留提交并根据工作流日志修复，不绕过现有发布流程。
